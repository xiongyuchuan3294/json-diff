# 规则配置设计

## 1. 设计目标

规则层的目标是将“如何比较”从程序中抽离出来，避免每个接口都写死逻辑，从而达到：

- 降低误报率
- 减少后续维护成本
- 支持不同接口使用不同对比策略
- 支持规则逐步演进而不重构系统主体


## 2. 规则适用范围

建议规则支持三个层级：

1. 全局规则
2. 接口规则
3. 场景规则

优先级建议：

- 场景规则 > 接口规则 > 全局规则


## 3. 规则分类

### 3.1 请求侧规则

请求侧规则的作用是保证“配对稳定”，而不是影响响应比较。

当前场景下，请求侧规则建议尽量少，只保留：

- path 标准化规则
- query 参数排序规则
- request body 规范化规则
- 特殊请求体序列化规则

说明：

- 当前已确认“同样 path + 同样请求参数”才算同一请求
- 因此请求参数本身不建议轻易做忽略
- 除非明确某些字段是纯噪声字段，否则不建议在请求侧忽略实际业务参数

### 3.2 响应侧规则

响应侧规则用于回答“两个版本的响应是不是业务一致”。

主要包括：

- 比较目标节点规则
- 忽略字段规则
- 数组比较规则
- 值等价规则
- 差异分级规则


## 4. 规则模型

### 4.1 比较目标节点规则

用于指定比较整包响应还是只比较某个节点。

常见模式：

- 比较整包响应
- 只比较 `$.data`
- 只比较 `$.result`

建议：

- 对有统一响应包裹结构的接口，优先比较 `$.data`
- 外层如 `timestamp`、服务端耗时、trace 字段可交给忽略规则处理

### 4.2 忽略字段规则

用于定义动态字段、噪声字段或无业务意义字段。

常见示例：

- `$.timestamp`
- `$.traceId`
- `$.requestId`
- `$.serverTime`

建议：

- 全局规则只放高度通用的动态字段
- 接口特有动态字段放在接口规则中
- 场景临时字段放在场景规则中

### 4.3 数组比较规则

建议支持三种模式：

- `ORDERED`：按顺序比较
- `UNORDERED`：忽略顺序比较
- `BY_KEY`：按业务主键比较

推荐优先级：

- 优先 `BY_KEY`
- 其次 `UNORDERED`
- 最后 `ORDERED`

### 4.4 值等价规则

用于处理“值不同但业务可视为相同”的情况。

建议支持：

- `50000` 与 `50000.0` 等价
- `null` 与缺失字段是否等价
- 空字符串与 `null` 是否等价
- `null` 与 `[]` 是否等价

说明：

- 这些规则必须按接口控制，不建议全局一刀切

### 4.5 差异分级规则

建议分为：

- `SAME`
- `IGNORABLE`
- `NORMAL`
- `BLOCK`

推荐示例：

- `status_code != 200`：`BLOCK`
- `retCode != 0`：`BLOCK`
- `$.data.totalCount` 变化：`NORMAL` 或 `BLOCK`
- `$.timestamp` 变化：`IGNORABLE`


## 5. 规则字段建议

建议在 `t_compare_rule` 中描述以下内容：

- 基本信息：规则编码、规则名称、作用域、优先级
- 命中条件：`sysid`、接口 path、`scenario_id`
- 比较目标：目标 JSON Path
- 忽略规则：忽略路径数组
- 数组规则：模式与 key 配置
- 值规则：等价关系配置
- 分级规则：路径与等级映射
- 生命周期：是否启用、生效时间范围


## 6. 规则命中顺序

建议命中流程如下：

1. 先查场景规则
2. 再查接口规则
3. 最后查全局规则
4. 将命中的规则合并为本次规则快照

### 6.1 合并原则

建议采用以下原则：

- 目标节点：高优先级覆盖低优先级
- 忽略路径：高低优先级合并去重
- 数组规则：高优先级覆盖低优先级
- 值等价规则：按路径维度覆盖
- 差异分级规则：按路径维度覆盖


## 7. 核心接口规则示例

以 `custTransInfo` 为例，建议规则如下：

### 7.1 基础规则

- 适用接口：`/aml/wst/custTransInfo`
- 比较目标：`$.data`
- 全局忽略：`$.timestamp`

### 7.2 数组规则

- 数组路径：`$.data.content`
- 比较模式：`BY_KEY`
- 业务主键：`transactionkey`

### 7.3 差异分级建议

- `$.data.totalPage` 变化：`NORMAL`
- `$.data.totalCount` 变化：`NORMAL`
- `$.data.content[*]` 核心字段变化：`NORMAL` 或 `BLOCK`
- 外层 `success`、`retCode` 变化：`BLOCK`


## 8. 规则治理建议

### 8.1 规则变更流程

建议规则变更遵循以下流程：

1. 提出规则变更申请
2. 说明变更原因和适用接口
3. 评估是否影响历史判定口径
4. 审核通过后生效
5. 必要时对历史任务补跑验证

### 8.2 规则快照建议

建议在 `t_compare_result` 中保存本次命中的规则快照，而不是只保存规则 ID。

这样做的好处：

- 即使后续规则变更，也能解释历史结果
- 有助于排查“为什么这次结果和上次不同”

### 8.3 规则数量控制

建议避免为每个接口配置过多碎片化规则。优先策略是：

- 全局规则尽量少但稳定
- 接口规则聚焦关键接口
- 场景规则只用于临时补充


## 9. 规则治理风险

### 9.1 忽略过多

如果忽略字段过多，会导致真正的差异被掩盖。

### 9.2 忽略过少

如果忽略字段过少，会导致大量误报，影响结果可信度。

### 9.3 规则漂移

如果规则变更没有沉淀快照，历史结果将难以解释。

### 9.4 接口规则缺失

如果关键接口没有定义数组主键或核心字段分级，结果会不稳定。


## 10. 推荐落地顺序

建议先做以下规则：

1. 全局动态字段忽略规则
2. 核心接口目标节点规则
3. 核心数组主键规则
4. 差异分级规则
5. 值等价规则

先解决“可用”和“误报低”，再逐步提升规则精细度。
