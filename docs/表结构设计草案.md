# 表结构设计草案

## 1. 设计目标

在不修改 `t_request_info` 表结构的前提下，通过新增支撑表完成以下能力：

- 为请求生成可配对的请求指纹
- 建立回归任务边界
- 存储接口比对规则
- 存储比对汇总结果
- 存储字段级差异明细


## 2. 设计原则

### 2.1 原表只读

`rrs_test_dev.t_request_info` 继续作为原始事实表使用：

- 不新增字段
- 不调整索引
- 不改变现有写入逻辑

### 2.2 派生能力外置

所有新增的“标准化、配对、规则、结果”能力统一落在新增表中，避免影响原始采集链路。

### 2.3 结果可回溯

所有新增表都应能通过 `request_info_id` 或 `trace_id` 回溯到 `t_request_info` 中的原始报文。


## 3. 新增表总览

建议新增以下表：

1. `t_request_compare_index`：请求索引表
2. `t_regression_batch`：回归任务表
3. `t_compare_rule`：比对规则表
4. `t_compare_result`：比对结果汇总表
5. `t_compare_result_detail`：比对结果明细表


## 4. `t_request_compare_index`

### 4.1 表职责

该表用于保存从 `t_request_info` 派生出的标准化请求信息，是“请求配对”的核心基础表。

该表解决的问题：

- `t_request_info` 不能加字段
- 请求 path、参数需要标准化后才能稳定配对
- 请求指纹需要落库以支持快速匹配

### 4.2 字段建议

| 字段名 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint | 是 | 主键 |
| `request_info_id` | bigint | 是 | 对应 `t_request_info.id` |
| `trace_id` | varchar(64) | 是 | 冗余请求流水号 |
| `sysid` | varchar(64) | 否 | 系统标识 |
| `scenario_id` | varchar(255) | 否 | 样本集合标识 |
| `env_tag` | varchar(64) | 否 | 环境标识 |
| `method` | varchar(16) | 是 | HTTP 方法 |
| `normalized_path` | varchar(1024) | 是 | 标准化后的接口路径 |
| `normalized_query` | longtext | 否 | 标准化后的 query 参数 |
| `normalized_body` | longtext | 否 | 标准化后的请求体 |
| `normalized_request_params` | longtext | 否 | 标准化后的请求参数组合 |
| `request_fingerprint` | varchar(128) | 是 | 请求指纹 |
| `request_hash` | varchar(128) | 否 | 请求摘要 |
| `response_hash` | varchar(128) | 否 | 原始响应摘要，便于快速定位 |
| `parse_status` | varchar(32) | 是 | 标准化状态 |
| `parse_error_msg` | varchar(1024) | 否 | 标准化失败原因 |
| `created_time` | datetime | 是 | 创建时间 |
| `update_time` | datetime | 是 | 更新时间 |

### 4.3 关键说明

- `normalized_request_params = normalized_query + normalized_body`
- `request_fingerprint = hash(sysid + method + normalized_path + normalized_request_params)`
- `response_hash` 仅用于辅助定位，不参与请求配对
- `parse_status` 建议取值：`SUCCESS`、`FAILED`、`SKIPPED`

### 4.4 索引建议

建议索引：

- 唯一索引：`request_info_id`
- 唯一索引：`trace_id`
- 普通索引：`scenario_id + request_fingerprint`
- 普通索引：`sysid + method + normalized_path`
- 普通索引：`parse_status`

### 4.5 数据来源

由标准化任务从 `t_request_info` 抽取并生成。


## 5. `t_regression_batch`

### 5.2 字段建议

| 字段名 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |

| `batch_code` | varchar(64) | 是 | 任务编码 |
| `batch_name` | varchar(255) | 是 | 任务名称 |
| `biz_name` | varchar(128) | 否 | 业务名称 |
| `old_scenario_id` | varchar(255) | 是 | 旧版本场景 |
| `new_scenario_id` | varchar(255) | 是 | 新版本场景 |
| `status` | varchar(32) | 是 | 任务状态 |
| `total_sample_count` | bigint | 否 | 总样本数 |
| `pairable_count` | bigint | 否 | 可配对数 |
| `matched_count` | bigint | 否 | 实际配对成功数 |
| `same_count` | bigint | 否 | 完全一致数 |
| `diff_count` | bigint | 否 | 存在差异数 |
| `only_old_count` | bigint | 否 | 仅旧版存在数 |
| `only_new_count` | bigint | 否 | 仅新版存在数 |
| `invalid_count` | bigint | 否 | 无效样本数 |
| `block_count` | bigint | 否 | 阻断差异数 |
| `operator` | varchar(64) | 否 | 发起人 |
| `start_time` | datetime | 否 | 开始时间 |
| `end_time` | datetime | 否 | 结束时间 |
| `remark` | varchar(1024) | 否 | 备注 |
| `created_time` | datetime | 是 | 创建时间 |
| `update_time` | datetime | 是 | 更新时间 |

### 5.3 状态建议

建议状态值：

- `INIT`
- `RUNNING`
- `PARTIAL_SUCCESS`
- `SUCCESS`
- `FAILED`
- `CANCELED`

### 5.4 索引建议

- 唯一索引：`batch_code`
- 普通索引：`old_scenario_id`
- 普通索引：`new_scenario_id`
- 普通索引：`status`


## 6. `t_compare_rule`

### 6.1 表职责

该表用于描述接口回归规则，是“减少误报”和“统一比较策略”的配置来源。

### 6.2 字段建议

| 字段名 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint | 是 | 主键 |
| `rule_code` | varchar(64) | 是 | 规则编码 |
| `rule_name` | varchar(255) | 是 | 规则名称 |
| `scope_type` | varchar(32) | 是 | 作用域：GLOBAL / API / SCENARIO |
| `sysid` | varchar(64) | 否 | 系统标识 |
| `api_path` | varchar(1024) | 否 | 适用接口路径 |
| `scenario_id` | varchar(255) | 否 | 适用场景 |
| `target_json_path` | varchar(1024) | 否 | 比较目标节点 |
| `ignore_paths` | longtext | 否 | 忽略字段路径集合 |
| `array_compare_mode` | varchar(32) | 否 | 数组比较模式 |
| `array_key_map` | longtext | 否 | 数组主键配置 |
| `value_equivalence_rule` | longtext | 否 | 值等价规则 |
| `severity_rule` | longtext | 否 | 差异分级规则 |
| `priority` | int | 是 | 优先级 |
| `enabled` | tinyint | 是 | 是否启用 |
| `effective_start_time` | datetime | 否 | 生效开始时间 |
| `effective_end_time` | datetime | 否 | 生效结束时间 |
| `creator` | varchar(64) | 否 | 创建人 |
| `created_time` | datetime | 是 | 创建时间 |
| `update_time` | datetime | 是 | 更新时间 |

### 6.3 关键说明

- `ignore_paths`、`array_key_map`、`value_equivalence_rule`、`severity_rule` 建议采用 JSON 存储
- 规则优先级建议：场景级高于接口级，高于全局规则
- 若一个接口未命中专属规则，则退回全局规则

### 6.4 索引建议

- 唯一索引：`rule_code`
- 普通索引：`scope_type + priority`
- 普通索引：`sysid + api_path`
- 普通索引：`scenario_id`
- 普通索引：`enabled`


## 7. `t_compare_result`

### 7.1 表职责

该表用于保存一次请求配对后的汇总结果，是任务查询和报表统计的主表。

### 7.2 字段建议

| 字段名 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint | 是 | 主键 |
| `batch_id` | bigint | 是 | 所属回归任务 |
| `old_scenario_id` | varchar(255) | 是 | 旧版场景 |
| `new_scenario_id` | varchar(255) | 是 | 新版场景 |
| `request_fingerprint` | varchar(128) | 否 | 请求指纹 |
| `sysid` | varchar(64) | 否 | 系统标识 |
| `method` | varchar(16) | 否 | HTTP 方法 |
| `api_path` | varchar(1024) | 否 | 标准化接口路径 |
| `old_request_info_id` | bigint | 否 | 旧版原始记录 ID |
| `new_request_info_id` | bigint | 否 | 新版原始记录 ID |
| `old_trace_id` | varchar(64) | 否 | 旧版 trace_id |
| `new_trace_id` | varchar(64) | 否 | 新版 trace_id |
| `pair_status` | varchar(32) | 是 | 配对状态 |
| `compare_status` | varchar(32) | 是 | 比较状态 |
| `diff_level` | varchar(32) | 否 | 差异等级 |
| `diff_count` | int | 否 | 差异数量 |
| `rule_snapshot` | longtext | 否 | 本次使用规则快照 |
| `summary` | varchar(2000) | 否 | 差异摘要 |
| `compare_time` | datetime | 否 | 比较完成时间 |
| `created_time` | datetime | 是 | 创建时间 |
| `update_time` | datetime | 是 | 更新时间 |

### 7.3 建议状态值

`pair_status` 建议包括：

- `MATCHED`
- `ONLY_OLD`
- `ONLY_NEW`
- `MULTI_OLD`
- `MULTI_NEW`
- `INVALID_SAMPLE`

`compare_status` 建议包括：

- `PENDING`
- `SKIPPED`
- `SUCCESS`
- `FAILED`

`diff_level` 建议包括：

- `SAME`
- `IGNORABLE`
- `NORMAL`
- `BLOCK`

### 7.4 索引建议

- 普通索引：`batch_id + request_fingerprint`
- 普通索引：`batch_id + compare_status`
- 普通索引：`batch_id + diff_level`
- 普通索引：`old_trace_id`
- 普通索引：`new_trace_id`
- 普通索引：`api_path`


## 8. `t_compare_result_detail`

### 8.1 表职责

该表用于保存字段级差异，适合做问题定位、结果展示和人工复核。

### 8.2 字段建议

| 字段名 | 类型建议 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint | 是 | 主键 |
| `result_id` | bigint | 是 | 对应 `t_compare_result.id` |
| `batch_id` | bigint | 是 | 所属任务 |
| `json_path` | varchar(2048) | 是 | 差异路径 |
| `diff_type` | varchar(32) | 是 | 新增 / 删除 / 修改 |
| `old_value` | longtext | 否 | 旧值 |
| `new_value` | longtext | 否 | 新值 |
| `severity` | varchar(32) | 否 | 严重级别 |
| `rule_source` | varchar(255) | 否 | 命中的规则来源 |
| `is_ignored` | tinyint | 是 | 是否忽略 |
| `sort_no` | int | 否 | 展示排序 |
| `created_time` | datetime | 是 | 创建时间 |

### 8.3 索引建议

- 普通索引：`result_id`
- 普通索引：`batch_id`
- 普通索引：`severity`
- 普通索引：`json_path(255)`


## 9. 表间关系

建议关系如下：

- `t_request_info` 1 对 1 `t_request_compare_index`
- `t_regression_batch` 1 对多 `t_compare_result`
- `t_compare_result` 1 对多 `t_compare_result_detail`
- `t_compare_rule` 与结果表不做强外键，可通过规则快照弱关联

这样设计的优点是：

- 原始事实与派生能力解耦
- 任务与结果清晰分层
- 后续规则变更不会直接污染历史结果解释


## 10. 数据生命周期建议

### 10.1 原始数据

- 来源：`t_request_info`
- 生命周期：以采集系统既有策略为准

### 10.2 索引数据

- 来源：由原始表派生
- 生命周期：建议与原始数据基本一致或略长

### 10.3 结果数据

- 汇总结果建议保留更久，便于做趋势统计
- 明细结果可根据容量设置独立保留周期


## 11. 推荐落地顺序

建议按照以下顺序建表：

1. 先建 `t_request_compare_index`
2. 再建 `t_regression_batch`
3. 再建 `t_compare_rule`
4. 然后建 `t_compare_result`
5. 最后建 `t_compare_result_detail`

这样可以先跑通索引和配对，再逐步补齐规则与结果能力。
