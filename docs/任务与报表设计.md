# 任务与报表设计

## 1. 设计目标

本文档聚焦于“系统如何被使用”和“结果如何被消费”，重点描述：

- 回归任务如何创建与执行
- 回归状态如何流转
- 失败项如何补跑
- 报表如何汇总与展示
- 运维和使用方最关心哪些指标


## 2. 任务输入设计

创建一次回归任务时，建议输入以下信息：

- 任务名称
- 业务名称或系统名称
- `old_scenario_id`
- `new_scenario_id`
- 规则集范围
- 操作人
- 备注

说明：

- 样本范围由 `old_scenario_id` 和 `new_scenario_id` 决定
- 规则可采用“默认规则集 + 场景补充规则”的方式


## 3. 任务状态流转

### 3.1 建议状态

建议状态包括：

- `INIT`
- `RUNNING`
- `PARTIAL_SUCCESS`
- `SUCCESS`
- `FAILED`
- `CANCELED`

### 3.2 状态含义

- `INIT`：任务已创建，尚未开始
- `RUNNING`：任务正在扫描、配对或执行对比
- `PARTIAL_SUCCESS`：任务整体完成，但存在失败或无效样本
- `SUCCESS`：任务整体完成且无失败项
- `FAILED`：任务在关键阶段失败，无法输出有效结果
- `CANCELED`：任务被人工取消

### 3.3 状态转换建议

建议状态转换如下：

- `INIT -> RUNNING`
- `RUNNING -> SUCCESS`
- `RUNNING -> PARTIAL_SUCCESS`
- `RUNNING -> FAILED`
- `INIT -> CANCELED`
- `RUNNING -> CANCELED`


## 4. 任务执行过程设计

建议从使用者视角将任务分成以下阶段：

1. 创建任务
2. 检查样本数据
3. 执行索引补齐
4. 请求配对
5. 响应对比
6. 结果汇总
7. 查看差异
8. 决定是否补跑

### 4.1 样本检查

在任务启动前，建议先做样本前置检查：

- 旧版场景是否存在数据
- 新版场景是否存在数据
- 两边索引是否齐全
- 是否存在明显重复样本

### 4.2 结果检查

任务结束后，建议至少提供：

- 一致样本数量
- 差异样本数量
- 单边缺失样本数量
- 阻断差异样本数量
- 无效样本数量


## 5. 补跑设计

### 5.1 补跑场景

建议支持以下补跑方式：

- 按整个任务补跑
- 按某个接口 path 补跑
- 按某个 `request_fingerprint` 补跑
- 按失败项补跑
- 按无效样本补跑

### 5.2 补跑触发时机

典型触发时机包括：

- 规则调整后需要重新验证
- 某批次样本采集不完整后补采
- `DeepDiff` 执行失败后重试
- 某个关键接口结果需要单独复核

### 5.3 补跑结果处理

建议：

- 补跑结果保留新的执行记录
- 主任务结果可更新汇总统计
- 历史执行痕迹应保留，便于审计


## 6. 报表设计

### 6.1 批次总览报表

建议展示：

- 任务名称
- `old_scenario_id`
- `new_scenario_id`
- 任务状态
- 总样本数
- 可配对数
- 完全一致数
- 差异数
- 阻断差异数
- 单边缺失数
- 任务耗时

### 6.2 接口维度报表

建议按接口 path 聚合展示：

- 接口 path
- 样本数
- 一致数
- 差异数
- 差异率
- 阻断差异数
- 最近一次差异时间

### 6.3 差异明细报表

建议支持按以下维度筛选：

- 任务
- 接口 path
- 差异等级
- `trace_id`
- `request_fingerprint`
- `json_path`

建议明细展示：

- 差异路径
- 旧值
- 新值
- 差异类型
- 命中规则
- 旧版样本标识
- 新版样本标识

### 6.4 单条结果详情页

建议支持查看：

- 原始请求摘要
- 请求 path 和请求参数
- 旧版响应摘要
- 新版响应摘要
- 差异明细列表
- 命中的规则快照


## 7. 使用角色设计

建议至少考虑以下角色：

- 测试人员：发起任务、查看结果、补跑任务
- 开发人员：定位差异原因、查看明细
- 管理人员：查看批次总体通过率和接口差异率
- 管理员：维护规则、处理权限、查看系统运行状态


## 8. 统计口径设计

### 8.1 样本总数

建议定义为：旧版和新版场景下所有进入任务范围的样本总量。

### 8.2 可配对数

建议定义为：在两边场景中能通过 `request_fingerprint` 建立唯一配对的请求数。

### 8.3 一致数

建议定义为：配对成功且 `response_body` 比较结果为无有效差异的数量。

### 8.4 差异数

建议定义为：配对成功且存在有效业务差异的数量。

### 8.5 阻断差异数

建议定义为：存在关键错误、核心结构变化或关键字段变化的数量。

### 8.6 单边缺失数

建议定义为：仅存在于旧版或仅存在于新版的样本数量。


## 9. 运维关注点

建议重点监控以下指标：

- 每日任务数
- 任务平均耗时
- `SUCCESS` / `FAILED` / `PARTIAL_SUCCESS` 比例
- 索引构建失败率
- `DeepDiff` 执行失败率
- 单接口差异率异常波动
- 非 JSON 响应占比
- 重复样本数量

### 9.1 告警建议

建议对以下情况配置告警：

- 某任务失败
- 某任务阻断差异数异常升高
- 索引构建连续失败
- `DeepDiff` 执行异常率升高
- 结果落库异常


## 10. 运营与治理建议

### 10.1 数据保留

建议区分三类数据的保留周期：

- 原始数据
- 索引数据
- 结果数据

### 10.2 数据导出

建议导出能力支持：

- 批次汇总导出
- 接口差异导出
- 单条差异明细导出

同时建议：

- 导出默认脱敏
- 原始报文导出增加权限校验

### 10.3 复核机制

建议支持人工复核差异结果，例如：

- 标记已确认差异
- 标记误报
- 添加备注
- 记录复核人和复核时间

这样有利于后续持续优化规则。


## 11. 推荐落地顺序

建议优先落地以下能力：

1. 任务创建与执行
2. 批次汇总报表
3. 差异明细查询
4. 失败补跑
5. 规则运维与人工复核

这样可以先满足“能跑、能看、能定位”，再逐步扩展“能治理、能复盘”。
