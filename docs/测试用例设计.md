# 测试用例设计

## 1. 设计目标

本文档用于定义接口新旧版本回归系统的测试用例范围，为后续造数、自测和测试报告提供统一口径。

设计目标：

- 覆盖正常场景
- 覆盖请求配对异常场景
- 覆盖响应异常场景
- 覆盖数据质量异常场景
- 覆盖规则与结果判定场景


## 2. 用例设计原则

### 2.1 先覆盖主流程

主流程必须覆盖以下核心能力：

- 按 `scenario_id` 区分旧版与新版数据
- 按“同 path + 同请求参数”完成请求配对
- 配对后比较 `response_body`
- 生成汇总结果和字段级差异

### 2.2 异常场景单独建例

凡是可能导致误判、漏判、无法比较或任务失败的情况，都应作为独立测试用例设计，而不能只混在正常数据中顺带验证。

### 2.3 结果必须可判定

每个测试用例都需要提前定义预期结果，包括：

- 是否能够配对
- 是否允许进入比较
- 最终属于哪种结果状态
- 是否应被判定为“有问题的请求”


## 3. 问题请求定义

当前测试口径中，以下情况都应视为“有问题的请求”：

- 找不到配对请求
- 仅旧版存在或仅新版存在
- 要比较的请求任一侧 `status_code` 不是 `200`
- `response_body` 为空或不是合法 JSON
- 同一侧同一请求指纹出现多条候选样本，无法稳定选主样本
- 响应比较后存在阻断差异

因此测试报告中不应只统计“diff 了多少条”，还应统计“问题请求数量”和“问题请求类型分布”。


## 4. 用例分类

建议按五类组织测试用例：

1. 正常功能类
2. 配对异常类
3. 响应异常类
4. 数据质量异常类
5. 规则判定类


## 5. 正常功能类用例

### TC001 正常配对且响应完全一致

- 场景：旧版和新版 `path`、请求参数完全一致
- 响应：两边 `response_body` 结构和值一致，仅 `timestamp` 不同
- 预期：
  - 可成功配对
  - 忽略动态字段后判定为一致
  - 不计入问题请求

### TC002 正常配对且存在一般差异

- 场景：旧版和新版请求完全一致
- 响应：`$.data.totalCount`、`$.data.totalPage` 发生变化
- 预期：
  - 可成功配对
  - 差异可识别
  - 计入问题请求
  - 差异等级为 `NORMAL`

### TC003 正常配对且存在阻断差异

- 场景：旧版和新版请求完全一致
- 响应：`success`、`retCode` 或核心结构发生明显变化
- 预期：
  - 可成功配对
  - 差异等级为 `BLOCK`
  - 计入问题请求

### TC004 数组顺序不同但业务一致

- 场景：`$.data.content` 元素一致，但顺序不同
- 规则：按业务主键比对
- 预期：
  - 可成功配对
  - 不应误判为差异
  - 不计入问题请求

### TC005 数组主键相同但字段值变化

- 场景：`$.data.content` 中同一 `transactionkey` 的元素字段值变化
- 预期：
  - 可成功配对
  - 应定位到字段级差异
  - 计入问题请求


## 6. 配对异常类用例

### TC101 旧版存在，新版找不到配对请求

- 场景：旧版场景中存在请求，新版场景中不存在相同 `path + 请求参数`
- 预期：
  - 无法配对
  - `pair_status = ONLY_OLD`
  - 计入问题请求

### TC102 新版存在，旧版找不到配对请求

- 场景：新版场景中存在请求，旧版场景中不存在相同 `path + 请求参数`
- 预期：
  - 无法配对
  - `pair_status = ONLY_NEW`
  - 计入问题请求

### TC103 同 path 但请求参数不同，不应错误配对

- 场景：接口路径相同，但 `request_body` 或 `query_params` 不同
- 预期：
  - 不应配成一对
  - 应分成两个不同请求指纹
  - 若对端不存在对应样本，则计入问题请求

### TC104 同一侧同一请求指纹出现重复样本

- 场景：同一个 `scenario_id` 下出现多条相同请求指纹
- 预期：
  - 系统能够识别冲突
  - 若能选主样本则继续比较，并记录冲突信息
  - 若无法选主样本，则计入问题请求

### TC105 `scenario_id` 写错导致跨批次混入

- 场景：旧版或新版样本被错误写入其他批次 `scenario_id`
- 预期：
  - 当前任务无法正确配对对应样本
  - 计入问题请求
  - 报告中应能定位为样本边界问题


## 7. 响应异常类用例

### TC201 任一侧状态码不是 200

- 场景：旧版或新版 `status_code != 200`
- 预期：
  - 即使请求能配对，也应判定为问题请求
  - 优先判为 `BLOCK`
  - 可选择不进入 JSON diff，或进入前先标记异常

### TC202 双方状态码都不是 200

- 场景：旧版、新版都失败
- 预期：
  - 仍属于问题请求
  - 应在结果中保留两边状态码
  - 不应误判为“一致通过”

### TC203 `response_body` 为空

- 场景：任一侧响应体为空字符串或 NULL
- 预期：
  - 判定为问题请求
  - `compare_status` 不应标记为正常成功

### TC204 `response_body` 不是合法 JSON

- 场景：任一侧返回 HTML、纯文本或格式错误 JSON
- 预期：
  - 判定为问题请求
  - 标记为非结构化响应或解析失败

### TC205 一侧 JSON，另一侧非 JSON

- 场景：旧版返回 JSON，新版返回 HTML 错误页
- 预期：
  - 判定为问题请求
  - 差异等级为 `BLOCK`

### TC206 JSON 结构根节点类型变化

- 场景：旧版返回对象，新版返回数组或 `null`
- 预期：
  - 判定为问题请求
  - 差异等级为 `BLOCK`


## 8. 数据质量异常类用例

### TC301 请求体不是合法 JSON

- 场景：`request_body` 本应为 JSON，但存入了非法字符串
- 预期：
  - 索引构建标记失败或降级处理
  - 该样本计入问题请求

### TC302 query 参数格式异常

- 场景：`query_params` 不是合法 JSON 或字段缺失
- 预期：
  - 索引构建可失败或降级
  - 需要在结果中标记数据质量问题

### TC303 关键字段缺失

- 场景：`url`、`method`、`trace_id` 等关键字段为空或异常
- 预期：
  - 无法正常生成请求指纹
  - 计入问题请求

### TC304 `response_body` 超大

- 场景：返回内容很大，包含大量数组元素
- 预期：
  - 系统仍可完成对比
  - 不应因单条大报文导致任务整体失败

### TC305 动态字段未忽略导致误报

- 场景：仅 `timestamp`、`requestId` 等动态字段变化
- 预期：
  - 配置正确时不计入问题请求
  - 若规则缺失，应能通过测试暴露误报问题


## 9. 规则判定类用例

### TC401 只比较 `$.data`

- 场景：外层字段变化但 `$.data` 一致
- 规则：目标节点只比较 `$.data`
- 预期：
  - 若外层字段在忽略范围内，则不计入问题请求

### TC402 忽略字段规则生效

- 场景：`$.timestamp`、`$.traceId` 变化
- 预期：
  - 差异被忽略
  - 不计入问题请求

### TC403 数组按主键比较规则生效

- 场景：数组元素顺序变化，元素内容相同
- 规则：`BY_KEY`
- 预期：
  - 不计入问题请求

### TC404 差异分级规则生效

- 场景：`retCode` 变化
- 预期：
  - 判为 `BLOCK`
  - 计入问题请求

### TC405 值等价规则生效

- 场景：`50000` 与 `50000.0` 对比
- 预期：
  - 若配置为等价，则不计入问题请求


## 10. 推荐首批落地的测试数据范围

如果先做一轮最小可用验证，建议首批至少覆盖以下 10 个用例：

- `TC001` 正常一致
- `TC002` 一般差异
- `TC003` 阻断差异
- `TC004` 数组顺序变化但一致
- `TC101` 仅旧版存在
- `TC102` 仅新版存在
- `TC104` 重复样本冲突
- `TC201` 状态码非 200
- `TC204` 响应非 JSON
- `TC305` 动态字段误报防护

这样既能覆盖主流程，也能覆盖你特别强调的“问题请求”。


## 11. 测试报告口径建议

测试报告建议至少包含以下统计项：

- 总测试用例数
- 通过用例数
- 失败用例数
- 问题请求数
- 问题请求分类分布
- 误报用例数
- 漏报用例数

其中“问题请求分类分布”建议至少拆成：

- 找不到配对请求
- 状态码异常
- 响应体为空
- 响应非 JSON
- 重复样本冲突
- 结构化差异


## 12. 后续造数建议

后续在 MySQL 中造测试数据时，建议：

- 每个测试用例至少落一对或一组对应数据
- `scenario_id` 使用统一批次，例如：
  - `custTransInfo#old#20260309_01`
  - `custTransInfo#new#20260309_01`
- `trace_id` 保持唯一，且命名中可带用例编号，便于回溯
- 样本数据中显式区分“正常样本”和“异常样本”
