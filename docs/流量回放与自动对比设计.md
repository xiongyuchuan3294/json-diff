# P0 流量回放 + 自动 Diff 实施方案

## Summary
在保留现有“直接对比（scenario/trace）”能力不变的前提下，新增“按条件回放并自动对比”模式，支持：
- 按 `scenario_id` 回放
- 按 `scenario_id + api_path` 回放
- 按 `trace_id` 条件回放  
并在回放完成后自动执行现有 diff（`源场景 vs 回放场景`），失败请求按“继续回放并汇总失败”策略处理。

## Key Changes
### 1. 公共接口（CLI）扩展（保持兼容）
- 现有命令不变；仅在显式开启 `--replay` 时进入回放模式。
- 回放模式新增参数：
  - `--replay`：开启回放
  - `--replay-target-base-url`：统一目标 BaseURL（必填）
  - 选择器二选一：
    - `--replay-source-scenario-id`（可配 `api_paths`/`--fuzzy`）
    - `--replay-trace-ids`（逗号分隔）
  - 节奏参数（可选）：`--replay-speed-factor`、`--replay-min-gap-ms`、`--replay-max-gap-ms`、`--replay-timeout-ms`、`--replay-retries`
- 返回结果新增：`replay_batch_id`、`replay_scenario_id`、`replay_stats`、`regression_batch_id`、`report_path`。

### 2. 回放筛选、排序与请求重建
- 数据源：`t_request_info`（`deleted=0`）。
- 筛选逻辑：
  - `scenario_id` 模式：按场景取样；若给 path 则按标准化 path 过滤（支持 `--fuzzy`）。
  - `trace_id` 模式：按 trace 精确命中。
- 排序逻辑（保证“按序回放”）：`start_time ASC, start_time_ms ASC, id ASC`。
- 请求重建：
  - 保留原 `method/query/body/headers` 主体
  - URL 重写为：`target_base_url + 原 path + 原 query`
  - 头处理：移除 hop-by-hop/`host`/`content-length`，其余透传。

### 3. 回放执行与限速方案（P0 严格顺序）
- 并发固定为 `1`（严格串行，优先顺序稳定性）。
- 节奏算法：
  - `recorded_gap = 当前样本start_ts - 上一样本start_ts`
  - `planned_gap = clamp(recorded_gap / speed_factor, min_gap_ms, max_gap_ms)`
  - 每次发起前 sleep `planned_gap`
- 默认值：
  - `speed_factor=1.0`
  - `min_gap_ms=300`
  - `max_gap_ms=3000`
  - `timeout_ms=10000`
  - `retries=1`（指数退避）
- 失败策略：
  - 不中断整体回放，继续执行后续请求
  - 超时/异常写入合成响应（建议 `status_code=599` + 错误 JSON），确保后续 diff 可感知为阻断差异。

### 4. 数据落库与可追溯性（已确认：新增回放表）
- 新增 `t_replay_batch`：
  - 记录回放任务元数据：选择器、目标地址、节奏参数、状态、统计、开始/结束时间等。
- 新增 `t_replay_request`：
  - 记录每条回放明细：`seq_no`、源 `request_info_id/trace_id`、回放 `request_info_id/trace_id`、HTTP结果、耗时、错误信息等。
- 回放响应写回 `t_request_info`：
  - 新 `scenario_id` 统一为 `源场景#replay#时间戳`
  - 新 `trace_id` 生成可追溯格式（如 `RP_<batch>_<seq>`）
  - 不修改 `t_request_info` 表结构。

### 5. 回放后自动调用现有 Diff
- 回放结束后自动调用现有回归流程：
  - `old_scenario_id = source_scenario_id`
  - `new_scenario_id = replay_scenario_id`
  - `api_paths` 继承回放筛选范围（trace 模式下从样本反推 path 集）
- 统计口径沿用当前实现：
  - `diff_level=BLOCK` => `compare_status=FAILED`
  - 其余保持现有逻辑
- `--replay` 未开启时，现有直接对比路径完全不变（兼容保证）。

### 6. 文档与初始化脚本落地
- 新增文档：`docs/流量回放与自动对比设计.md`（本方案正文落地）
- 更新目录：`docs/设计文档目录.md`
- 更新初始化 SQL：`sql/init_rrs_test_dev.sql`
  - 增加回放表 DDL（`CREATE TABLE IF NOT EXISTS`）

## Test Plan
1. 选择器与预检
- `scenario_id`、`scenario_id+path`、`trace_id` 三种筛选命中正确。
- 无数据、trace 不存在、trace 跨场景（P0 直接失败）等预检行为正确。

2. 顺序与节奏
- 同一批样本实际发送顺序与排序键一致。
- 实际间隔满足 `planned_gap`（允许小范围调度误差）。
- 默认节奏下不会“瞬时打满”目标接口。

3. 回放结果落库
- `t_replay_batch` 状态流转与统计正确。
- `t_replay_request` 明细与源/回放 `request_info` 关联正确。
- 回放失败请求也有明细并可追踪错误。

4. 自动 Diff 联动
- 回放完成后自动产生回归批次和报告。
- 部分失败时仍执行 diff，且阻断口径正确。
- 与现有直接对比模式回归测试全部通过。

5. 兼容性回归
- 原 `scenario` 对比命令行为不变。
- 原 `trace pair` 对比命令行为不变。
- 现有报表输出结构不回退。

## Assumptions
- 回放目标采用“统一目标 BaseURL”（已确认）。
- 回放后比较采用“源场景 vs 回放场景”（已确认）。
- 失败策略为“继续回放并汇总失败”（已确认）。
- P0 为保证顺序稳定性，回放并发固定为 1，不做并行回放。
- 不修改 `t_request_info` 结构，仅新增回放支撑表。
