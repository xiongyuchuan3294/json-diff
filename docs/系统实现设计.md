# 系统实现设计

## 1. 设计目标

本文档用于说明接口新旧版本回归系统的实现层设计，重点回答以下问题：

- 从原始请求数据到回归结果的处理链路如何组织
- 各模块各自负责什么
- 请求如何配对
- 响应如何调用 `DeepDiff` 进行 JSON diff
- 任务如何执行、失败如何重试、结果如何回写


## 2. 总体实现架构

建议将系统拆分为以下模块：

1. 请求索引构建模块
2. 回归任务编排模块
3. 请求配对模块
4. 响应标准化模块
5. `DeepDiff` 对比执行模块
6. 差异解析模块
7. 结果落库模块
8. 统计汇总模块
9. 统一执行入口模块（`runner`，供 CLI/Skill/MCP 复用）

可理解为一条处理链：

`原始请求 -> 索引构建 -> 批次任务 -> 请求配对 -> 响应标准化 -> DeepDiff diff -> 结果落库 -> 报表统计`


## 3. 模块设计

### 3.1 请求索引构建模块

#### 职责

- 从 `t_request_info` 读取原始数据
- 解析 URL、query 参数、request body
- 生成标准化 path 和标准化请求参数
- 生成 `request_fingerprint`
- 将结果写入 `t_request_compare_index`

#### 输入

- `t_request_info` 原始记录

#### 输出

- `t_request_compare_index` 索引记录

#### 注意事项

- 该模块应支持幂等执行
- 若一条记录已经构建过索引，应支持跳过或覆盖更新
- 若请求体不是合法 JSON，也要尽量降级处理，不能让全批任务失败

### 3.2 回归任务编排模块

#### 职责

- 创建并维护 `t_regression_batch`
- 指定 `old_scenario_id` 和 `new_scenario_id`
- 组织索引扫描、配对、对比、落库的执行顺序
- 更新任务状态与任务统计

#### 输入

- 回归任务参数
- 场景标识

#### 输出

- 一条或多条回归结果记录

#### 推荐方式

- 建议异步执行，不建议由前端长连接等待任务完成
- 支持手动触发和定时触发两种模式

### 3.3 请求配对模块

#### 职责

- 按 `old_scenario_id` 读取旧版样本索引
- 按 `new_scenario_id` 读取新版样本索引
- 基于 `request_fingerprint` 进行配对
- 识别单边缺失、重复样本、多候选样本

#### 配对原则

当前已确认的配对口径为：

- 同一个接口 path
- 同样的请求参数

因此：

- `response_body` 不参与配对
- 请求参数标准化后用于生成 `request_fingerprint`

#### 输出类型

- 成功配对
- 仅旧版存在
- 仅新版存在
- 一侧多样本冲突
- 索引无效

### 3.4 响应标准化模块

#### 职责

- 根据接口规则读取需要比较的响应节点
- 删除动态字段
- 统一对象排序
- 处理数值和空值等价规则
- 对数组按策略对齐或预处理

#### 设计要点

- 标准化必须发生在 `DeepDiff` 对比之前
- 标准化规则必须可配置，不建议写死在逻辑中
- 若某接口只比较 `$.data`，应在该模块完成截取

### 3.5 `DeepDiff` 对比执行模块

#### 职责

- 将标准化后的旧版与新版响应输出为临时 JSON
- 调用 `DeepDiff(ignore_order=True)` 完成结构化 diff
- 接收 diff 输出并传给差异解析模块

#### 设计要点

- `DeepDiff` 只负责对比，不负责业务规则判断
- 对比执行应当与业务逻辑解耦，便于后续替换对比引擎
- 若某一对样本无法进入 `DeepDiff`，应在结果中标记失败原因

### 3.6 差异解析模块

#### 职责

- 解析 `DeepDiff` 输出
- 转换成统一的差异明细结构
- 映射 `json_path`、差异类型、旧值、新值、严重级别
- 生成结果摘要

#### 输出

- 一条结果汇总
- 多条差异明细

### 3.7 结果落库模块

#### 职责

- 写入 `t_compare_result`
- 写入 `t_compare_result_detail`
- 回写任务统计

#### 设计要点

- 结果落库建议分两步：先写汇总，再写明细
- 如明细落库失败，至少应保留汇总结果和错误信息

### 3.8 统计汇总模块

#### 职责

- 统计每个任务的可配对数、一致数、差异数、阻断数
- 统计每个接口的差异率
- 生成报表页需要的聚合结果


## 4. 关键处理链路

### 4.1 索引构建链路

建议链路：

1. 从 `t_request_info` 取待处理原始记录
2. 解析 `url`
3. 提取 path 和 query 参数
4. 解析 `request_body`
5. 生成 `normalized_path`
6. 生成 `normalized_request_params`
7. 生成 `request_fingerprint`
8. 写入 `t_request_compare_index`

### 4.2 回归执行链路

建议链路：

1. 创建回归任务
2. 检查旧版与新版场景的索引数据是否齐全
3. 扫描两边索引数据
4. 生成配对关系
5. 逐对读取原始 `response_body`
6. 做响应标准化
7. 调用 `DeepDiff`
8. 解析结果并落库
9. 汇总任务统计并更新任务状态

### 4.3 失败处理链路

建议链路：

1. 记录失败原因
2. 标记当前配对结果状态
3. 不中断整个任务
4. 最终将任务状态更新为 `PARTIAL_SUCCESS` 或 `FAILED`


## 5. 请求指纹设计

### 5.1 设计目标

请求指纹用于回答“这两条请求是不是同一个请求”。

### 5.2 组成要素

当前建议的请求指纹组成如下：

- `sysid`
- `method`
- `normalized_path`
- `normalized_request_params`

其中：

- `normalized_request_params = normalized_query + normalized_body`

### 5.3 重要边界

- `response_body` 不参与请求指纹生成
- `headers` 默认不参与请求指纹生成
- `host`、`port`、协议默认不参与请求指纹生成

### 5.4 主样本选择

若同一个 `scenario_id` 下出现多个相同 `request_fingerprint`，建议按以下顺序选主样本：

1. `status_code = 200`
2. `response_body` 是合法 JSON
3. `end_time` 最新
4. `id` 最大


## 6. 响应比较设计

### 6.1 比较对象

响应比较对象为配对完成后的：

- 旧版 `response_body`
- 新版 `response_body`

### 6.2 比较前处理

进入 `DeepDiff` 前，应完成以下处理：

- 提取目标比较节点
- 忽略动态字段
- 统一对象排序
- 按规则处理数组
- 统一数值和空值语义

### 6.3 比较结果类型

建议结果分为：

- 无差异
- 可忽略差异
- 一般差异
- 阻断差异
- 无法比较


## 7. 幂等、重试与补跑设计

### 7.1 幂等要求

以下动作建议具备幂等性：

- 索引构建
- 任务创建
- 单条配对结果写入
- 失败重试

### 7.2 重试策略

建议支持以下重试范围：

- 重试索引构建失败项
- 重试某一个任务
- 重试某一个接口路径
- 重试某一个 `request_fingerprint`
- 仅重试 `compare_status = FAILED` 的结果

### 7.3 补跑场景

以下情况适合补跑：

- 新增或调整了规则
- 某个接口的数组主键配置补齐了
- 某批次数据采集异常后补采
- `DeepDiff` 执行阶段因环境问题失败


## 8. 性能设计

### 8.1 性能瓶颈点

潜在瓶颈主要在：

- 索引构建时的 JSON 解析
- 回归任务中的大批量配对
- 大响应体的 `DeepDiff` 比较
- 差异明细的大量落库

### 8.2 优化建议

建议优先做以下优化：

- 先构建索引后再配对，避免运行时重复解析原始请求
- 配对时优先从索引表读取，而不是直接扫原始表
- 结果明细采用批量写入
- 报表统计尽量基于汇总表而不是明细表
- 大任务采用分页或分片处理

### 8.3 并发建议

建议：

- 任务级别串行，任务内部可并行
- 单任务内部按接口或按请求指纹分片并行
- 并发数应可配置，避免 `DeepDiff` 执行占满资源


## 9. 可观测性设计

建议记录以下运行指标：

- 索引构建成功数与失败数
- 每次任务的总耗时
- 每次任务的配对成功率
- 每次任务的 diff 条数
- 非 JSON 样本数量
- 重复样本数量

建议记录以下日志信息：

- 请求索引构建异常
- 配对冲突信息
- 规则命中信息
- `DeepDiff` 执行失败信息
- 结果落库失败信息


## 10. 安全与权限设计

建议考虑以下约束：

- 原始请求和响应可能包含敏感信息
- 不是所有人都应具备查看原始报文的权限
- 导出结果时应支持脱敏

因此建议：

- 原始数据查看权限与汇总结果查看权限分离
- 结果页默认展示脱敏值
- 明细导出增加权限控制


## 11. 推荐落地顺序

建议实现顺序如下：

1. 先完成 `t_request_compare_index` 构建链路
2. 再完成任务编排和请求配对
3. 然后完成响应标准化与 `DeepDiff` 集成
4. 再完成结果汇总和明细落库
5. 最后补充报表、补跑、监控与权限能力


## 12. MCP Tool 骨架（最小可用）

当前已提供最小骨架实现：

- 入口脚本：`scripts/regression_mcp_server.py`
- 传输方式：`stdio`
- 复用核心：通过 `runner.py` 复用索引构建、预检查、任务创建、回归执行、报告生成逻辑

已暴露工具：

- `run_api_regression(api_paths, old_scenario_id, new_scenario_id)`
- `run_api_regression_advanced(...)`
- `ping()`

其中：

- `api_paths=ALL` 或 `*` 表示按 `scenario_id` 下全部接口批量对比
- `api_paths=/a,/b` 表示多接口子集对比
- 默认报告路径：`output/<batch_code>/测试报告.md`，并写 `output/latest.md`
