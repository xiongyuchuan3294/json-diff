本文档用于汇总当前项目中的接口新旧版本回归设计文档，明确各文档的职责、阅读顺序和已确认约束，便于后续继续落地实现。

## 3. 建议阅读顺序

建议按照以下顺序阅读：

1. 先看 `接口新旧版本回归对比方案.md`
2. 再看 `表结构设计草案.md`
3. 然后看 `系统实现设计.md`
4. 再看 `规则配置设计.md`
5. 最后看 `任务与报表设计.md`

这样能够先建立总体理解，再进入结构、实现、规则和运行层面。


## 4. 当前已确认约束

### 4.1 原始事实表不能修改

已确认：

- `t_request_info` 的表结构不能修改
- 所有新增能力都必须通过新增表实现
- 原始请求与响应仍以 `t_request_info` 为唯一事实来源

### 4.2 版本样本集合由 `scenario_id` 标识

已确认 `scenario_id` 的写法为：

- `业务场景#版本侧标识#执行批次`

当前示例：

- `custTransInfo#old#20260309_01`
- `custTransInfo#new#20260309_01`

### 4.3 请求配对口径

已确认“同一个请求”的判定方式为：

- 同一个接口 path
- 同样的请求参数

其中请求参数包括：

- `query_params`
- `request_body`

### 4.4 响应比较口径

已确认：

- `response_body` 不参与请求配对
- 请求配对完成后，再对成对样本的 `response_body` 做 JSON diff
- JSON diff 引擎选型为 `DeepDiff(ignore_order=True)`


## 5. 文档之间的关系

可以将当前设计文档理解为四层：

- 总体方案：回答“要做什么、为什么这样做”
- 表结构设计：回答“新增哪些表、每张表做什么”
- 系统实现设计：回答“整体处理链路怎么跑”
- 规则与任务设计：回答“怎么配规则、怎么执行、怎么查看结果”


## 6. 进入实现前建议先确认的事项

在进入实际开发前，建议最后确认以下内容：

- 是否接受新增 4 到 5 张支撑表
- `scenario_id` 是否保证由回归任务统一生成
- 第一批核心接口有哪些
- 每个核心接口的数组主键字段是什么
- 差异分级是否按测试视角还是按业务影响定义
- 是否需要页面化展示结果，还是先以库表查询为主
