## 1. 背景与目标

### 1.1 背景

当前系统已经能够将接口请求与响应完整落库，样例表为 `rrs_test_dev.t_request_info`。这为接口回归提供了原始数据基础，但还缺少一套可执行、可复用、可扩展的“新旧版本自动对比方案”。

接口回归的核心问题不是简单比较两段文本，而是：

1. 如何判定两条记录属于“同一个请求”。
2. 如何忽略动态字段、噪声字段、环境差异。
3. 如何针对 JSON 结构做字段级差异分析。
4. 如何沉淀为批量执行、可追踪、可汇总的回归体系。

### 1.2 目标

建设一套接口新旧版本回归方案，实现以下目标：

- 自动识别相同请求参数的请求对。
- 自动比较旧版本和新版本响应 JSON 是否存在结构或字段差异。
- 支持忽略动态字段、忽略对象字段顺序、配置数组比对方式。
- 输出可追踪的差异明细和汇总结果。
- 支持按批次执行回归，便于版本切换、补跑和结果追溯。

### 1.3 非目标

本方案当前不覆盖以下内容：

- 不直接做接口压测或性能基准测试。
- 不直接替代接口自动化测试框架。
- 不尝试用数据库 SQL 直接完成复杂 JSON 结构对比。


## 2. 总体思路

整体思路分为四层：

1. **采集层**：保存原始请求、响应、上下文信息。
2. **配对层**：识别旧版本和新版本中的“同一请求”。
3. **对比层**：对响应内容做标准化处理，并调用 `DeepDiff(ignore_order=True)` 执行结构化 JSON diff。
4. **结果层**：将差异汇总、明细、统计信息落库，用于查询与报表展示。

一句话总结：

> 先配对，再标准化，后 diff，最后沉淀结果。


## 3. 为什么选择 `DeepDiff`

### 3.1 推荐原因

对于当前实现，选择 `DeepDiff` 作为 JSON 对比引擎，原因如下：

- 直接集成在 Python 代码中，环境搭建成本更低。
- 原生支持 `ignore_order=True`，适合列表顺序不稳定的接口响应。
- 对嵌套 JSON 的对象、数组、类型变化都有成熟 diff 能力。
- 与当前 MySQL 造数、自测脚本同属 Python 技术栈，便于快速落地和回归验证。

### 3.2 设计定位

`DeepDiff` 在本方案中的角色是 **JSON diff 引擎**，只负责回答以下问题：

- 两段已标准化的 JSON 有哪些字段差异。
- 差异位于哪些 JSON Path。
- 差异类型是新增、删除、修改还是类型变化。

以下问题不交给 `DeepDiff` 解决，而由业务系统负责：

- 哪两条记录应该进行比对。
- 哪些字段应该忽略。
- 哪些差异属于阻断，哪些属于可忽略。
- 配对失败、非 200、非 JSON 这些问题请求如何分类。


## 4. 数据现状分析

现有表 `t_request_info` 已经具备以下优势：

- 保存了原始 `url`、`method`、`headers`、`query_params`、`request_body`。
- 保存了 `status_code`、`response_body`、耗时、时间戳等信息。
- 保存了 `sysid`、`client_ip`、`trace_stack_md5` 等定位字段。

这张表适合作为“原始事实表”，继续保留，不建议为了对比功能而大幅改造原表语义。

当前约束已经明确：`t_request_info` 的表结构不能修改，因此新增能力必须通过新表承载，而不能通过给原表加字段、加索引来实现。

但若要支持新旧版本回归，还需要明确以下关键能力：

- 需要约定 `scenario_id` 的写法，用于区分旧版本和新版本样本。
- 若同一个 `scenario_id` 会被反复复用，还需要额外区分同一轮回归任务范围。
- 需要请求指纹，才能高效配对相同请求。
- 需要对比结果表，才能保存字段级 diff。


## 5. 比对对象定义

### 5.1 什么叫“同一个请求”

对于当前场景，建议将“同一个请求”定义为：**同一个接口 path，且请求参数一致**。

建议参与配对的字段为：

- `sysid`
- `method`
- 规范化后的 URL 路径
- 规范化后的请求参数

其中，请求参数包括：

- 规范化后的 `query_params`
- 规范化后的 `request_body`

可表示为：

`same_request = sysid + method + normalized_path + normalized_request_params`

其中：

`normalized_request_params = normalized_query + normalized_body`

需要特别说明：`response_body` 不参与 `same_request` 的计算，它只在请求配对完成后作为比对对象进入 JSON diff。

### 5.2 为什么不直接用原始 URL

原始 URL 中通常带有 host、port、协议等环境信息，例如测试环境、预发环境、旧版实例和新版实例地址可能不同。若直接用原始 URL 全字段比较，会导致同一业务请求无法跨环境配对。

因此建议：

- URL 只抽取路径部分参与匹配。
- query 参数单独提取并排序后参与匹配。
- host、port、scheme 默认不参与请求指纹。

### 5.3 headers 是否参与匹配

默认不参与。

只有在明确某些 header 会影响业务结果时，才将其纳入匹配因子，例如：

- 租户标识
- 渠道标识
- 用户语言
- 特定业务开关

以下字段建议默认忽略：

- `cookie`
- `user-agent`
- `referer`
- `content-length`
- `x-page-url`
- `origin`
- `accept-encoding`

### 5.4 基于样例的说明

结合现有样例：

- `id=3483` 与 `id=4377` 的 `method`、URL 路径、请求参数一致，应视为同一请求。
- `id=3565` 的请求参数中 `custId`、`caseDate` 不同，应视为另一个请求指纹，不应与前两条配对。


## 6. `scenario_id` 与批次管理设计

### 6.1 优先复用 `scenario_id`

既然现有表已经包含 `scenario_id`，并且后续可以由你写入不同值来区分新旧版本，那么建议优先复用该字段作为样本集合标识。

在一次回归任务中，可以直接指定：

- `old_scenario_id`
- `new_scenario_id`

系统分别查询两边样本后，再通过新增的请求索引表获取 `request_fingerprint` 并进行配对。

这样做的好处是：

- 不需要为了版本标识额外新增 `version_tag`
- 与当前表结构兼容，改造成本低
- 查询条件清晰，便于直接按场景筛数和补跑

### 6.2 已确认的 `scenario_id` 命名规范

`scenario_id` 已确认按以下规则写入：

- `业务场景#版本侧标识#执行批次`

当前示例为：

- `custTransInfo#old#20260309_01`
- `custTransInfo#new#20260309_01`

该规范中建议约定：

- 第一段表示业务场景，例如 `custTransInfo`
- 第二段表示版本侧标识，例如 `old`、`new`
- 第三段表示执行批次，例如 `20260309_01`

这样做的价值是：

- 能直接通过 `scenario_id` 区分旧版和新版样本集合
- 能保证同一业务场景多次回归时不混入历史数据
- 便于后续按场景、版本侧、批次做筛选和统计

在回归任务中，可直接指定：

- `old_scenario_id = custTransInfo#old#20260309_01`
- `new_scenario_id = custTransInfo#new#20260309_01`

### 6.3 建议新增请求索引表

由于 `t_request_info` 不能改表，建议新增一张请求索引表，例如 `t_request_compare_index`，专门承载回归所需的派生字段。

建议字段包括：

- `request_info_id`：关联 `t_request_info.id`
- `trace_id`：冗余存储，便于查询
- `sysid`
- `scenario_id`
- `env_tag`：环境标识，例如 `test`、`uat`
- `method`
- `normalized_path`：规范化路径
- `normalized_query`
- `normalized_body`
- `request_fingerprint`：请求指纹
- `request_hash`：规范化请求摘要
- `response_hash`：规范化响应摘要
- `parse_status`：规范化是否成功
- `created_time`
- `update_time`

其中：

- `scenario_id` 仍用于区分旧版样本集合与新版样本集合
- `request_fingerprint` 用于跨 `scenario_id` 配对
- `request_hash`、`response_hash` 用于快速去重与定位
- `request_info_id` / `trace_id` 用于把对比结果回溯到原始报文

### 6.4 建议新增回归任务表

建议新增 `t_regression_batch`，记录一轮回归任务的元信息。即使请求表中已经有 `scenario_id`，任务表仍然有价值，因为它负责显式记录“这次比较的是哪两个场景”。

建议包含：

- 批次 ID
- 批次名称
- `old_scenario_id`
- `new_scenario_id`
- 执行状态
- 发起人
- 开始时间
- 结束时间
- 备注信息


## 7. 请求规范化设计

### 7.1 规范化目标

规范化的目的是去除非业务性差异，使同一业务请求能够稳定命中同一个指纹。

### 7.2 规范化规则

#### URL 规范化

- 提取路径部分，例如 `/aml/wst/custTransInfo`
- 移除协议、host、port
- query 参数拆分为独立键值对
- query 参数按 key 排序

#### query 参数规范化

- 对 JSON 或 key-value 做统一序列化
- 字段按 key 排序
- 删除无意义空白字符

#### request body 规范化

- 对 JSON 请求体进行反序列化再序列化
- 对对象 key 排序
- 去掉无意义空格、换行
- 若 body 为表单，也要转换成稳定排序结构

#### headers 规范化

- 默认不参与请求指纹
- 如接口需要，可按白名单提取业务 header 参与指纹生成

### 7.3 请求指纹生成策略

建议使用规范化后的关键字段拼接后生成摘要：

`request_fingerprint = hash(sysid + method + normalized_path + normalized_request_params)`

其中：

`normalized_request_params = normalized_query + normalized_body`

请求指纹只用于请求配对，不用于响应比较。响应比较始终以配对完成后的 `response_body` 为输入。

这样做的好处：

- 能快速配对
- 能避免长文本直接比较
- 便于建立索引提升查询效率


## 8. 响应规范化设计

### 8.1 为什么不能直接比较原始 response_body

接口响应中通常包含大量动态字段，例如：

- 时间戳
- 随机流水号
- traceId
- requestId
- 服务端生成的临时标识

如果直接拿原始 JSON 做 diff，将产生大量误报，导致回归结果不可用。

### 8.2 响应规范化规则

建议在进入 `DeepDiff` 之前先做一次响应标准化处理：

- 删除可忽略动态字段
- 统一对象字段顺序
- 对数值类型做一致化处理
- 必要时只截取业务有效节点，例如只比较 `$.data`

### 8.3 建议默认忽略的动态字段

建议支持全局默认忽略路径，例如：

- `$.timestamp`
- `$.traceId`
- `$.requestId`
- `$.cost`
- `$.serverTime`

同时支持按接口追加忽略路径。

### 8.4 数值与空值语义

建议将以下规则做成可配置项，而不是硬编码：

- `50000` 与 `50000.0` 是否等价
- `null` 与缺失字段是否等价
- `null` 与空数组 `[]` 是否等价
- 空字符串与 `null` 是否等价


## 9. 数组比对策略设计

数组是接口回归中最容易误判的部分，必须重点设计。

### 9.1 三种推荐策略

#### 策略一：按顺序比较

适用于返回列表顺序稳定且顺序本身有业务意义的场景。

优点：

- 简单直观
- 实现成本低

缺点：

- 一旦顺序变化，会产生大量误报

#### 策略二：忽略顺序比较

适用于列表元素相同，但返回顺序不稳定的场景。

优点：

- 能减少顺序导致的误报

缺点：

- 当元素内容相似时，最优匹配逻辑会更复杂

#### 策略三：按业务主键比较

适用于数组元素中存在稳定主键的场景，例如：

- `transactionkey`
- `id`
- `code`

建议优先使用该方式，因为它最贴近业务语义。

### 9.2 结合样例的建议

你的样例中：

- `$.data.content` 是交易列表。
- 每个元素包含 `transactionkey`。

因此推荐：

- 对 `$.data.content` 按 `transactionkey` 对齐再比较。

这样即使顺序变化，也不会误判成整段列表差异。


## 10. 差异分级设计

仅判断“有差异/无差异”不足以支持回归决策，建议引入差异分级。

### 10.1 建议分级

- `一致`：无任何有效差异。
- `可忽略差异`：仅动态字段或配置忽略项发生变化。
- `一般差异`：存在业务字段变化，但不一定阻断上线。
- `阻断差异`：接口失败、核心结构变化、关键业务字段变化。

### 10.2 常见分级示例

- `status_code` 从 `200` 变为 `500`：阻断差异
- `retCode` 从 `0` 变为非 `0`：阻断差异
- `$.data.totalCount` 变化：一般差异或阻断差异，取决于接口定义
- `$.timestamp` 变化：可忽略差异
- `$.data.content[transactionkey=xxx].custName` 变化：一般差异

为保证回归批量执行性能，建议重点在新增表上建立索引，而不是修改 `t_request_info`：

- 在 `t_request_compare_index` 上建立 `scenario_id + request_fingerprint`
- 在 `t_request_compare_index` 上建立 `sysid + method + normalized_path`
- 在 `t_request_compare_index` 上建立 `request_info_id` 或 `trace_id` 唯一索引
- 在 `t_compare_result` 上建立 `batch_id + request_fingerprint`
- 在 `t_compare_result` 上建立 `compare_status`
- 在 `t_compare_result` 上建立 `old_trace_id`
- 在 `t_compare_result` 上建立 `new_trace_id`


## 12. 回归执行流程

### 12.1 全流程

建议执行流程如下：

1. 接口请求与响应原始落库。
2. 写入有区分度的 `scenario_id`，分别代表旧版样本集合和新版样本集合。
3. 从 `t_request_info` 抽取数据，写入 `t_request_compare_index`，完成规范化并生成 `request_fingerprint`。
4. 在回归任务中指定 `old_scenario_id` 与 `new_scenario_id`，通过索引表按“同 path + 同请求参数”的请求指纹进行配对。
5. 配对完成后，仅针对成对样本的 `response_body` 读取接口规则并做标准化。
6. 将标准化后的旧版 `response_body` 与新版 `response_body` 交给 `DeepDiff(ignore_order=True)` 执行 diff。
7. 解析 diff 结果并落汇总表与明细表。
8. 输出批次级统计、接口级统计、差异清单。

### 12.2 配对策略

配对时建议按以下优先级：

- 先根据 `old_scenario_id` 和 `new_scenario_id` 划定两边样本集合
- 同一 `request_fingerprint` 必须唯一配对
- 若同一 `scenario_id` 下同指纹存在多条记录，应先约定主样本选择策略

### 12.3 重复记录处理

对于同一 `scenario_id`、同一请求指纹出现多条样本，建议定义固定策略，例如：

- 优先选 `status_code = 200` 的记录
- 其次选 `end_time` 最新的一条
- 或保留全部，但标记一条为主记录参与回归


## 13. `DeepDiff` 集成方案

### 13.1 集成方式

建议采用以下模式：

- 使用程序或脚本从数据库中取出一对已配对的响应数据。
- 根据规则完成响应标准化。
- 在 Python 代码中直接调用 `DeepDiff(ignore_order=True)` 执行对比。
- 将 `DeepDiff` 输出转换成统一的差异明细结构并回写数据库。

### 13.2 优势

相比依赖外部二进制工具，这种方式更适合当前项目，原因包括：

- Python 内直接调用，部署和运行环境更简单。
- 与当前原型、自测脚本、MySQL 造数脚本保持同一技术栈。
- 更容易在代码里叠加忽略路径、状态码判定、非 JSON 判定等业务逻辑。

### 13.3 注意事项

在使用 `DeepDiff` 前必须完成以下动作：

- 清理动态字段
- 统一对象排序或标准化
- 明确是否启用 `ignore_order=True`
- 明确值等价规则

否则 `DeepDiff` 输出也会包含大量业务无意义差异。


## 14. 输出结果设计

### 14.1 汇总结果

每一对请求应产出一条汇总结果，至少包括：

- 是否一致
- 差异级别
- 差异数量
- 关键摘要
- 旧记录 trace_id
- 新记录 trace_id

### 14.2 明细结果

每一处差异应记录：

- `json_path`
- 差异类型：新增 / 删除 / 修改
- 旧值
- 新值
- 严重级别
- 命中的规则

### 14.3 批次统计

建议输出以下批次级指标：

- 可比总数
- 完全一致数
- 存在差异数
- 仅旧版存在数
- 仅新版存在数
- 阻断差异数
- 各接口差异率


## 15. MVP 分阶段落地建议

### 15.1 第一阶段：最小可用版本

目标：尽快跑通一条完整回归链路。

建议范围：

- 复用现有 `scenario_id` 区分旧版和新版样本
- 新增 `t_request_compare_index` 承载 `request_fingerprint`
- 按请求指纹自动配对
- 只比较 `response_body` 或 `response_body.data`
- 支持全局忽略动态字段
- 使用 `DeepDiff(ignore_order=True)` 输出基础 diff
- 保存汇总结果

### 15.2 第二阶段：提升准确性

建议增强：

- 数组按业务主键对齐
- 接口级规则配置
- 差异分级
- 明细结果落库
- 批次级报表统计

### 15.3 第三阶段：平台化

建议增强：

- 规则配置页面
- 差异结果查询页面
- 批次对比报告页面
- 支持白名单 / 黑名单字段管理
- 支持人工确认差异与结果复核


## 16. 风险与注意事项

### 16.1 误判风险

如果忽略字段规则不完善，会出现大量误报；如果忽略过多，又可能漏报。

因此建议：

- 先从全局默认规则开始
- 再逐步补充接口级规则

### 16.2 配对错误风险

如果请求指纹定义过于简单，可能把不同请求错误地配成一对；若定义过于严格，则可能同一请求配不上。

因此建议：

- 首先使用 `method + path + query + body` 作为主匹配条件
- headers 默认不参与，仅白名单参与

### 16.3 数组误判风险

列表字段如果按下标对比，极易误判。建议尽量为关键数组指定业务主键。

### 16.4 `scenario_id` 污染风险

如果 `scenario_id` 命名过于粗，例如长期只写 `old` 和 `new`，回归结果会混入历史数据。因此建议 `scenario_id` 自带执行批次或日期信息，或在结果层引入独立任务批次表。


## 17. 结合当前样例的应用说明

以你提供的样例为例：

- `3483` 与 `4377` 会命中同一个 `request_fingerprint`。
- 如果两者分别落在 `old_scenario_id` 与 `new_scenario_id` 对应的数据集中，就应被配成一对。
- 对比前应忽略 `$.timestamp`。
- 然后重点比较：
  - `$.data.pageNum`
  - `$.data.pageSize`
  - `$.data.totalPage`
  - `$.data.totalCount`
  - `$.data.content`
- 对 `$.data.content` 推荐按 `transactionkey` 对齐比较。

从业务角度看，这一组样例中的核心差异不是时间戳不同，而是：

- 总页数变化
- 总记录数变化
- 列表元素数量变化
- 列表内容变化

这正说明结构化 JSON diff 比简单字符串比较更适合该场景。


## 18. 规则配置设计细化

### 18.1 规则分层

为了兼顾通用性与准确性，建议规则分为三层，按优先级从低到高覆盖：

1. **全局默认规则**：适用于所有接口，例如忽略 `$.timestamp`。
2. **接口级规则**：适用于某一类 URL 或某个接口路径。
3. **场景级规则**：适用于某次特殊回归任务，例如某一阶段临时忽略特定字段。

推荐的覆盖顺序是：

- 先应用全局规则
- 再应用接口级规则
- 最后应用场景级规则

这样既能减少重复配置，也能保留特殊场景的灵活性。

### 18.2 规则建议包含的维度

建议每条规则至少包含以下维度：

- 规则名称
- 适用范围：全局 / 接口 / 场景
- 适用接口路径
- 比较目标节点，例如整包响应或仅 `$.data`
- 忽略字段路径列表
- 数组比对策略
- 数组业务主键
- 空值等价规则
- 数值等价规则
- 差异分级规则
- 是否启用
- 生效时间

### 18.3 推荐的规则样式

对于大多数接口，规则可以概括为以下几类：

| 规则类型 | 用途 | 示例 |
| --- | --- | --- |
| 目标节点规则 | 缩小比较范围 | 只比较 `$.data` |
| 忽略路径规则 | 过滤动态字段 | 忽略 `$.timestamp` |
| 数组规则 | 统一列表比较方式 | `$.data.content` 按 `transactionkey` |
| 值等价规则 | 降低语义误判 | `50000` 与 `50000.0` 视为一致 |
| 严重级别规则 | 指导回归判定 | `retCode` 变化判为阻断 |

### 18.4 推荐的规则优先级示例

以 `custTransInfo` 接口为例，可按以下逻辑执行：

- 先应用全局忽略路径：`$.timestamp`
- 再应用接口规则：比较目标只看 `$.data`
- 再应用数组规则：`$.data.content` 按 `transactionkey` 配对
- 最后应用严重级别规则：`$.data.totalCount` 变化记为一般差异

这样可以把“如何比较”从程序里抽离出来，后续新增接口时只要配规则即可。


## 19. 异常样本处理方案

### 19.1 需要覆盖的异常类型

回归数据不一定都是标准成功样本，建议系统显式处理以下异常情况：

- `response_body` 为空
- `response_body` 不是合法 JSON
- `status_code` 非 `200`
- 请求超时或调用失败
- 同一请求指纹在某一侧没有匹配样本
- 同一请求指纹在某一侧出现多条候选样本

### 19.2 建议的处理策略

| 异常类型 | 是否进入 diff | 建议结果状态 |
| --- | --- | --- |
| 旧版无样本 | 否 | 仅新版存在 |
| 新版无样本 | 否 | 仅旧版存在 |
| 双方都有但响应为空 | 否 | 样本无效 |
| 响应非 JSON | 否或降级 | 非结构化响应 |
| `status_code` 不一致 | 可直接判定 | 阻断差异 |
| 同侧重复样本 | 选主样本后进入 | 待确认或已去重 |

### 19.3 非 JSON 响应的处理建议

虽然当前重点是 JSON 接口，但实际场景中仍可能遇到：

- 纯文本响应
- HTML 错误页
- 文件流或下载响应

建议策略如下：

- 若明确接口应返回 JSON，则此类数据直接标记为阻断差异。
- 若接口本身允许文本响应，则降级为字符串对比，但单独标识“非 JSON 比对”。
- 这类结果不要与 JSON 结构化 diff 混在一起统计。

### 19.4 主样本选择策略建议

当同一 `scenario_id`、同一 `request_fingerprint` 存在多条数据时，建议按以下顺序选主样本：

1. 优先 `status_code = 200`
2. 优先 `response_body` 为合法 JSON
3. 优先 `end_time` 最新
4. 如仍冲突，按 `id` 最大值

同时建议保留“候选样本数量”字段，便于后续排查重复采集问题。


## 20. 回归任务与状态流转设计

### 20.1 为什么要显式管理任务状态

即使已经用 `scenario_id` 标识新旧版本样本，也仍建议在结果层保留任务概念。因为一次回归不仅仅是“比较两个场景”，还包括：

- 创建任务
- 扫描待配对样本
- 执行对比
- 生成结果
- 人工复核
- 补跑失败项

### 20.2 推荐状态流转

建议任务状态至少包括：

- `INIT`：任务已创建，尚未执行
- `RUNNING`：正在扫描或对比中
- `PARTIAL_SUCCESS`：部分成功，存在失败或无效样本
- `SUCCESS`：全部完成
- `FAILED`：执行失败
- `CANCELED`：人工取消

### 20.3 建议保留的任务统计字段

建议在 `t_regression_batch` 中增加以下统计字段，方便直接展示任务结果：

- 总样本数
- 可配对数
- 完全一致数
- 一般差异数
- 阻断差异数
- 仅旧版存在数
- 仅新版存在数
- 无效样本数
- 执行耗时

### 20.4 补跑策略

建议支持以下补跑方式：

- 按整个任务重跑
- 按单个接口重跑
- 按指定 `request_fingerprint` 重跑
- 仅重跑失败项或无效样本项

这样有利于在规则调整后快速验证效果，而不必全量重做。


## 21. 验收口径与上线标准

### 21.1 技术验收口径

建议从以下维度验收方案是否可用：

- **配对准确率**：同请求是否能稳定配成一对
- **误报率**：动态字段是否被有效过滤
- **漏报率**：关键业务字段变化是否能被识别
- **执行效率**：批量回归是否能在可接受时间内完成
- **可追踪性**：能否从差异结果追溯到原始请求与响应

### 21.2 业务验收口径

建议让业务或测试同学参与验收以下结果：

- 关键接口差异是否能被准确识别
- 差异摘要是否足够直观
- 明细路径是否便于定位问题
- 报表是否能回答“哪些接口变了、变在哪、影响多大”

### 21.3 MVP 验收标准建议

建议第一阶段至少达到以下标准后再进入下一阶段：

- 能指定一对 `old_scenario_id` / `new_scenario_id` 完成一次完整回归
- 能稳定配对相同请求
- 能忽略全局动态字段
- 能输出结构化差异清单
- 能区分一致、差异、单边缺失三类结果
- 能针对至少一个核心接口完成数组按主键比对


## 22. 运维与治理建议

### 22.1 数据保留策略

由于请求与响应正文可能较大，建议对原始数据和结果数据分别制定保留策略：

- 原始请求响应：保留较短周期，例如 30 至 90 天
- 结果汇总数据：保留更长周期，便于趋势分析
- 结果明细数据：根据存储成本与查询频率决定周期

### 22.2 敏感信息治理

请求头、请求体、响应体中可能包含敏感数据，例如：

- 用户标识
- 证件号
- 账号
- Cookie 或 Session 信息

建议在回归平台层面增加以下治理措施：

- 结果展示时脱敏
- 明确谁可以查看原始报文
- 敏感头字段默认不参与展示和导出
- 如有必要，对敏感字段进行存储加密

### 22.3 性能与容量建议

随着数据量增长，建议重点关注以下问题：

- `request_fingerprint` 相关索引是否命中
- 大字段表是否需要冷热分离
- 回归任务是否需要分页扫描或分片执行
- 结果明细是否需要异步落库

### 22.4 可观测性建议

建议对回归任务增加基础监控：

- 任务成功率
- 平均执行时长
- 单任务 diff 数量
- 单接口差异率
- 非 JSON 响应占比
- 主样本冲突数量

这样可以帮助快速判断问题出在数据采集、规则配置还是 diff 执行阶段。


## 23. 待确认事项

### 23.1 需要尽快拍板的事项

为了让方案顺利进入实现阶段，建议优先确认以下事项：

- `scenario_id` 的命名规范是否统一由回归任务生成
- 是否新增 `t_request_compare_index` 作为请求指纹索引表
- 是否新增 `t_regression_batch` 作为任务主表
- 是否新增 `t_compare_result` 与 `t_compare_result_detail` 保存结果
- 是否允许只比较 `$.data`，还是某些接口必须比较整包响应
- 核心接口的数组主键字段分别是什么
- `null`、空字符串、空数组之间的等价规则是否需要统一
- 差异分级是否以测试视角为主，还是以业务影响为主

### 23.2 建议的确认顺序

建议按以下顺序推进确认：

1. 先确认 `scenario_id` 规则与回归任务边界
2. 再确认结果表是否需要单独建设
3. 然后确认核心接口的规则清单
4. 最后确认报表、查询页面和复核流程

这样可以避免一开始就在平台层面投入过多实现成本。


## 24. 结论

本方案建议采用：

- **原始请求事实表 `t_request_info` 保留不动，且不改表结构**
- **优先复用 `scenario_id`，并通过新增请求索引表补充请求指纹能力做稳定配对**
- **新增规则层处理忽略字段与数组策略**
- **使用 `DeepDiff(ignore_order=True)` 作为 JSON 结构化对比引擎**
- **新增结果汇总与明细表支撑查询、统计和报告**

推荐的落地顺序是：

1. 先约定 `scenario_id` 命名规则，并建设请求索引表补齐请求指纹。
2. 再跑通 `DeepDiff(ignore_order=True)` 对比链路。
3. 然后补上接口级规则与数组主键比对。
4. 最后建设报表与平台化能力。

这样能够以最小成本快速落地，同时为后续扩展保留空间。
